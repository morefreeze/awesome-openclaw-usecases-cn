# 自主教育游戏开发流水线

## 痛点
**起源故事:** 一位"老派局域网玩家"爸爸想要为他的女儿苏珊娜(3岁)和即将出生的朱丽叶创建一个安全、无广告、高质量的游戏门户。现有的网站充斥着垃圾信息、攻击性广告和欺骗性按钮(暗黑模式),让他的幼儿感到沮丧。

**挑战:** 建立一个"干净、快速、简单"的门户是容易的部分。真正的挑战是填充**40多款适合特定发展阶段(0-15岁)的教育游戏**,而无需一个开发团队。对于一个单亲开发者来说,手动开发太慢了,而且在数十款游戏之间保持一致性正成为一场噩梦。

## 功能介绍
这个用例定义了一个"游戏开发者代理",可以自主管理游戏创建和维护的整个生命周期。工作流执行**"先修复漏洞"**策略,代理必须在实现新功能之前检查并解决报告的漏洞。

**效率:** 这个流水线能够**每7分钟生成一款新游戏或修复一个漏洞**。代理不知疲倦地迭代41款以上计划中的游戏,交替创建新内容和纠正之前周期中发现的问题。

当路径清晰时,代理会:
1.  **选择**: 基于"循环"策略从队列(`development-queue.md`)中识别下一款游戏,以平衡不同年龄段的内容。
2.  **实现**: 为游戏编写HTML5/CSS3/JS代码,遵循严格的`game-design-rules.md`(无框架、移动优先、离线可用)。
3.  **注册**: 自动将游戏元数据添加到中央注册表(`games-list.json`)。
4.  **文档**: 更新`CHANGELOG.md`和`master-game-plan.md`状态。
5.  **部署**: 处理Git工作流:获取主分支、创建功能分支、使用规范提交信息提交更改,并合并回主分支。

## 提示词

这个工作流的核心是给代理的**系统指令**。这个提示词将大语言模型变成一个遵守项目严格结构的 disciplined 开发者。

*(**注意:** 生产环境中使用的实际提示词是**西班牙语**(`es-419`),以符合项目的目标受众(拉丁美洲儿童)和该地区潜在的未来贡献者。下面的版本是为了这份文档而翻译的。)*

```text
担当网页游戏开发和儿童用户体验专家。
你的目标是开发生产队列中的下一款游戏。

请在开始前阅读并分析以下上下文文件:

1.  漏洞上下文(最高优先级 - 关键):
    @[bugs/]
    (检查这个文件夹。如果有文件,你的任务是修复**按字母顺序排列的第一个文件**。暂时忽略其余的漏洞和游戏队列。)

2.  队上下文(下一款游戏是哪个):
    @[development-queue.md]
    (在"下一款游戏"部分中识别标记为[NEXT]的游戏。仅当没有漏洞时。)

3.  设计规则(技术标准):
    @[game-design-rules.md]
    (严格遵循这些规则:纯HTML/CSS/JS、文件夹结构、移动端响应式)

4.  游戏规格(机制和资源):
    (根据游戏ID在games-backlog/中识别相应的文件)

5.  中央注册表(集成):
    @[public/js/games-list.json]
    (你必须在这个文件中注册新游戏,使其显示在首页)

任务:
0. **先修复漏洞!**: 如果`bugs/`文件夹有内容,你唯一的优先级是修复**按字母顺序排列的第一个漏洞**。创建一个`fix/...`分支,解决**那个**漏洞,更新状态,并合并。**不要尝试一次修复多个漏洞。**
   - 如果没有漏洞,继续下一款游戏:

1. **同步**: `git fetch && git pull origin master`(关键)。
2. 创建新分支: `git checkout -b feature/[game-id]`。
3. 在'public/games/[game-id]/'中创建文件夹和文件。
4. 根据待办事项和设计规则实现逻辑和设计。
5. 在'games-list.json'中注册游戏(关键)。
6. 完成后:
    - 更新`CHANGELOG.md`并升级版本。
    - 更新`master-game-plan.md`和`development-queue.md`。
    - 记录更改: `git commit -m "feat: add [game-id]"`。
7. **交付**:
    - 推送: `git push origin feature/[game-id]`。
    - 请求合并到主分支。
    - 一旦进入主分支,推送更改(`git push origin master`)。
```

## 所需技能
-   **Git**: 管理分支、提交和合并。

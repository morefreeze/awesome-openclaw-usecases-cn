# 多智能体专业团队(独立创始人设置)

独立创始人要承担所有角色——战略、开发、营销、销售、运营。在这些角色之间切换会破坏深度工作。招聘既昂贵又缓慢。如果你能组建一个小型的、专业的AI代理团队，每个代理都有不同的角色和个性，都可以通过单一聊天界面控制，那会怎么样？

这个用例将多个OpenClaw代理设置为一个协调的团队，每个代理都专门研究一个领域，通过共享内存进行通信，并通过Telegram进行控制。

## 痛点

- **一个代理无法做好所有事情**：当同时处理战略、代码、营销研究和业务分析时，单个代理的上下文窗口会很快填满
- **缺乏专业化**：通用提示产生通用输出——编码代理不应该同时撰写营销文案
- **独立创始人倦怠**：你需要的是一个团队，而不是另一个需要管理的工具。代理应该在后台工作并呈现结果，而不需要持续监督
- **知识孤岛**：营销研究的见解不会自动影响开发优先级，除非你手动连接它们

## 功能介绍

- **专业代理**：每个代理都有不同的角色、个性和针对其领域优化的模型
- **共享内存**：项目文档、目标和关键决策对所有代理都可见——不会丢失任何东西
- **私有上下文**：每个代理还维护自己的对话历史和特定领域的笔记
- **单一控制平面**：所有代理都可以通过一个Telegram群聊访问——标记你需要的代理
- **每日计划任务**：代理主动工作，无需被要求——内容提示、竞争对手监控、指标跟踪
- **并行执行**：多个代理可以同时处理独立任务

## 示例团队配置

### 代理1：Milo(战略负责人)

```text
## SOUL.md — Milo

你是Milo，团队负责人。自信、有大局观、有魅力。

职责：
- 战略规划和优先级排序
- 协调其他代理
- 每周目标设定和OKR跟踪
- 将所有代理的见解综合为可操作的决策

模型：Claude Opus
渠道：Telegram(响应@milo)

日常任务：
- 8:00 AM：查看代理的夜间活动，发布晨间站会摘要
- 6:00 PM：每日结束时总结每周目标的进展
```

### 代理2：Josh(业务与增长)

```text
## SOUL.md — Josh

你是Josh，业务分析师。务实、直截了当、以数据为导向。

职责：
- 定价策略和竞争分析
- 增长指标和KPI跟踪
- 收入建模和单位经济效益
- 客户反馈分析

模型：Claude Sonnet(快速、善于分析)
渠道：Telegram(响应@josh)

日常任务：
- 9:00 AM：提取并总结关键指标
- 每周跟踪竞争对手的定价变化
```

### 代理3：营销代理

```text
## SOUL.md — 营销代理

你是营销研究员。富有创造力、好奇心、了解趋势。

职责：
- 内容构思和起草
- 竞争对手社交媒体监控
- 跟踪Reddit/HN/X上的相关话题趋势
- SEO关键词研究

模型：Gemini(擅长网络研究和长上下文分析)
渠道：Telegram(响应@marketing)

日常任务：
- 10:00 AM：基于热门话题呈现3个创意
- 每天监控竞争对手在Reddit/X上的提及
- 每周内容日历草稿
```

### 代理4：开发代理

```text
## SOUL.md — 开发代理

你是开发代理。精确、彻底、有安全意识。

职责：
- 编码和架构决策
- 代码审查和质量检查
- 错误调查和修复
- 技术文档

模型：Claude Opus / Codex(用于实现)
渠道：Telegram(响应@dev)

日常任务：
- 检查CI/CD流水线健康状况
- 查看未合并的PR
- 标记技术债务项目
```

## 所需技能

- `telegram`技能用于共享控制界面
- `sessions_spawn` / `sessions_send`用于多代理协调
- 用于团队内存的共享文件系统或笔记工具
- 不同模型提供商的独立API密钥(如果使用混合模型)
- VPS或始终在线的机器来运行代理

## 设置方法

### 1. 共享内存结构

```text
team/
├── GOALS.md           # 当前OKR和优先级(所有代理都可以读取)
├── DECISIONS.md       # 关键决策日志(仅追加)
├── PROJECT_STATUS.md  # 当前项目状态(所有代理都可以更新)
├── agents/
│   ├── milo/          # Milo的私有上下文和笔记
│   ├── josh/          # Josh的私有上下文
│   ├── marketing/     # 营销代理的研究
│   └── dev/           # 开发代理的技术笔记
```

### 2. Telegram路由

配置一个所有代理都监听的单个Telegram组，但每个代理只有在被标记时才响应：

```text
## AGENTS.md — Telegram路由

Telegram组："Team"

路由：
- @milo → 战略代理(生成/恢复milo会话)
- @josh → 业务代理(生成/恢复josh会话)
- @marketing → 营销代理(生成/恢复marketing会话)
- @dev → 开发代理(生成/恢复dev会话)
- @all → 广播到所有代理
- 无标记 → 默认由Milo(团队负责人)处理

每个代理：
1. 读取共享的GOALS.md和PROJECT_STATUS.md获取上下文
2. 读取自己的私有笔记
3. 处理消息
4. 在Telegram中回复
5. 如果回复涉及决策或状态更改，更新共享文件
```

### 3. 计划任务

```text
## HEARTBEAT.md — 团队日程

每日：
- 8:00 AM：Milo发布晨间站会(汇总代理的夜间活动)
- 9:00 AM：Josh提取关键指标
- 10:00 AM：营销代理基于热门话题呈现创意
- 6:00 PM：Milo发布每日结束总结

持续：
- 开发：监控CI/CD健康状况，查看进来的PR
- 营销：Reddit/X关键词监控(每2小时)
- Josh：竞争对手价格检查(每周)

每周：
- 周一：Milo起草每周优先级(输入来自所有代理)
- 周五：Josh编译每周指标报告
```

## 关键见解

- **个性比你想象的更重要**：给代理不同的名字和沟通方式，让你很自然地“与团队交谈”，而不是与通用AI搏斗
- **共享内存+私有上下文**：这种组合至关重要——代理需要共同点(目标、决策)，但也需要自己的空间来积累领域专业知识
- **为合适的工作选择合适的模型**：不要使用昂贵的推理模型进行关键词监控。将模型能力与任务复杂度匹配
- **计划任务是飞轮**：当代理主动呈现见解时，真正的价值就会显现出来，而不仅仅是在你提问时
- **从2个开始，而不是4个**：从一个负责人+一个专家开始，然后在你发现瓶颈时添加代理

## 灵感来源

这种模式由[X上的Trebuh](https://x.com/iamtrebuh/status/2011260468975771862)描述，他是一位独立创始人，在VPS上设置了4个OpenClaw代理——Milo(战略负责人)、Josh(业务)、营销代理和开发代理——所有这些都通过单个Telegram聊天控制。每个代理都有自己的个性、模型和计划任务，同时共享项目内存。他将其描述为“一个真正的小团队，全天候可用”。

这种模式也在[OpenClaw Showcase](https://openclaw.ai/showcase)上得到了证实，`@jdrhyne`报告说运行“15+代理，3台机器，1个Discord服务器——大部分都是IT通过聊天构建的”，`@nateliason`描述了一个多模型流水线(原型→总结→优化→实现→重复)，每个阶段使用不同的模型。另一位用户`@danpeguine`运行两个不同的OpenClaw实例，在同一个WhatsApp组中协作。
